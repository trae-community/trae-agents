# 贡献指南

欢迎提交 PR！我们建议保持每个智能体"小而专"，专注于解决特定场景的问题。

![TRAE Agents Banner](./assets/image/Agents.gif)

## 🚀 创建新的智能体

### 1. Fork 仓库并创建分支
```bash
git checkout -b feature/your-agent-name
```

### 2. 创建智能体目录
- 在 `agents/` 目录下创建新文件夹
- 使用短横线命名法：`your-agent-name`
- 复制模板文件 `agents/_template/README.md` 到你的目录

### 3. 填写配置信息
根据模板要求填写以下内容：

#### ✅ 必需部分
- **基本信息**：名称、描述、适用场景
- **Prompt**：完整的系统指令
- **使用示例**：至少 1 个输入输出示例

#### 🔧 可选部分
- **MCP 服务配置**：如果需要 MCP
- **工具配置**：推荐的内置工具
- **协作智能体**：可以配合使用的其他智能体
- **配置建议**：模型选择、参数设置等
- **相关资源**：文档和参考链接

### 4. 测试你的智能体
在 TRAE 中实际创建并测试：
1. 打开 TRAE
2. 根据你的配置创建智能体
3. 使用不同的输入进行测试
4. 根据测试结果优化 Prompt

### 5. 更新 README
将你的智能体添加到 `README.md` 的智能体列表中：
```markdown
| your-agent-name | 一句话描述 | ✅ 稳定 | [配置详情](./agents/your-agent-name/) |
```

### 6. 提交 PR
- 确保所有文件都已提交
- 编写清晰的 commit message
- 推送到你的 fork
- 创建 Pull Request

## 📋 质量标准

你的智能体需要达到以下要求：

### ✅ 内容要求
- **定位清晰**：清楚说明这个智能体是做什么的
- **Prompt 完整**：包含角色定义、目标、工作流程、约束条件
- **示例实用**：展示真实的输入输出示例
- **可复现**：其他人按照你的配置能创建出同样效果的智能体

### ❌ 避免的问题
- 不要包含敏感信息（API Key、Token、内部 URL 等）
- 不要过于宽泛（一个智能体什么都做）
- 不要缺少示例
- 不要使用模糊的指令（比如"尽力而为"）

## 📁 文件结构

每个智能体目录应包含：

```
your-agent-name/
├── README.md          # 主要配置文件（必需）
└── resources/        # 相关资源文件（可选）
    └── reference.md
```

## 🏷️ 状态标识

在 README 中使用 emoji 标识智能体状态：

- 🚧 **开发中**：还在测试，可能会调整
- ✅ **稳定**：经过测试，效果良好
- ⚠️ **已弃用**：仍可用但不推荐用于新场景

## 🎯 命名规范

### 智能体名称
- 使用英文，首字母大写：`Git Commit Generator`
- 或使用短横线命名：`git-commit-generator`
- 明确表达用途，避免歧义

### 目录名称
- 必须使用短横线命名：`git-commit-generator`
- 全部小写
- 简洁明了

## ✅ PR 检查清单

提交前请确认：

- [ ] 创建了智能体目录和 README.md
- [ ] 包含完整的 Prompt（可以直接复制到 TRAE）
- [ ] 提供了至少 1 个使用示例
- [ ] 不包含敏感信息（API Key、Token 等）
- [ ] 更新了主 README 的智能体列表
- [ ] 在 TRAE 中实际测试过
- [ ] 遵循了模板结构
- [ ] 使用了正确的命名规范

## 💡 最佳实践

### 写好 Prompt 的技巧

1. **明确角色**："你是一个 XXX 专家"
2. **定义目标**："你的职责是..."
3. **分步骤**："第一步...第二步..."
4. **给示例**："例如..."
5. **说明约束**："不要..."

### 常见模式

#### 分析 + 建议型
```
你是一个代码审查专家。当用户提供代码时：
1. 分析代码结构和逻辑
2. 识别潜在问题
3. 提供改进建议
```

#### 任务执行型
```
你是一个 Git 助手。当用户需要写 commit message 时：
1. 阅读代码变更
2. 判断变更类型
3. 生成标准化的 commit message
```

#### 问答咨询型
```
你是一个技术咨询顾问。当回答用户技术问题时：
1. 理解问题核心
2. 提供多种解决方案
3. 分析每种方案的优缺点
```

## 🤔 常见问题

### Q: Prompt 应该多长？
A: 没有固定标准，通常 200-500 字即可。关键是清晰完整，不是越长越好。

### Q: 是否需要配置 MCP？
A: 大多数智能体不需要。只有在真正需要访问外部数据或服务时才使用。

### Q: 如何测试效果？
A: 在 TRAE 中创建后，用 3-5 个不同的真实场景测试，看是否都能给出合理回复。

### Q: 可以引用其他智能体吗？
A: 可以，但要确保被引用的智能体已经存在且稳定。
